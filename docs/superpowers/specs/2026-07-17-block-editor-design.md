# 블록 에디터 설계 — 노션풍 WYSIWYG 편집 (마크다운 저장)

2026-07-17 브레인스토밍 확정본. 페이지 작성 UI를 "제목 필드 + 작성/미리보기 탭 + 마크다운 textarea"에서
노션풍 WYSIWYG 블록 에디터로 재설계한다. 상위 맥락은 [플랫폼 로드맵](../../roadmap/2026-07-17-platform-roadmap.md) 참조.

## 범위

### 블록 화이트리스트 (마크다운 왕복 보장)

문단 · 제목 1~3 · 글머리 목록 · 번호 목록 · 체크박스 목록 · 인용 · 코드 블록(언어 선택, 복사 버튼) ·
구분선 · 표(GFM) · 이미지(URL 입력만 — 업로드는 파일 서비스 선행으로 로드맵 이관).

### 인라인 서식

굵게 · 기울임 · 취소선 · 인라인 코드 · 링크 · `[[위키링크]]`.

### 범위 밖 (로드맵 3단계, ⚠️ 저장 포맷 재논의)

토글, 콜아웃, 비디오, 임베드, 북마크, 데이터베이스 뷰, 밑줄, 글자색, Mention, 이모지 피커,
**텍스트 정렬**(GFM에 정렬 문법 없음 + MarkdownView가 raw HTML 미렌더 — 저장 포맷 재논의 필요 그룹).

## 아키텍처

### 신규 의존성

`@tiptap/react` + starter-kit + 표/체크리스트/링크/이미지/placeholder 확장 + suggestion 유틸 +
마크다운 직렬화 라이브러리. **정확한 패키지·버전은 구현 계획 첫 태스크에서 GFM 표/체크박스 왕복
변환을 스파이크로 검증한 뒤 고정한다.** 검증 실패 시 `prosemirror-markdown` 직접 구성으로 폴백.

### 디렉토리 구조

```
src/features/wiki/editor/
├── WikiEditor.tsx          # TipTap 래퍼. props: initialMarkdown, pages / ref: getMarkdown(), isDirty()
├── markdown.ts             # parseMarkdown(md) ↔ serializeMarkdown(doc) — 변환 단일 창구
├── extensions/
│   ├── wikiLink.ts         # [[제목]] 인라인 노드 + 자동완성 suggestion
│   └── slashMenu.tsx       # / 블록 삽입 메뉴
└── components/
    ├── BubbleToolbar.tsx   # 선택 시 플로팅 서식 툴바 (굵게/기울임/취소선/코드/링크)
    └── DragHandle.tsx      # 블록 앞 ⋮⋮ 핸들 — 블록 순서 드래그 이동
```

### 기존 코드 변화

- **PageEditPage**: Tabs(작성/미리보기) 제거 — WYSIWYG라 미리보기 불필요. 대형 인라인 제목 입력
  (placeholder "제목 없음") + WikiEditor 본문 + 저장/취소. 생성(?parent, ?title 프리필)/수정 공용 구조와
  라우트 리마운트 동작(`b388943`)은 유지.
- **PageViewPage**: 본문 렌더는 기존 `MarkdownView` 유지. 문서 폭·타이포만 에디터와 통일
  (중앙 정렬, 최대 폭 ~720px 문서 컬럼 — 뷰/편집 화면이 같은 레이아웃 규격 공유).
- **WikiLinkTextArea**: 삭제. 자동완성 로직(`activeLinkQuery`, 후보 8개 제한, `[`·`]`·개행 제목 제외)은
  `extensions/wikiLink.ts`로 이식.
- **스토어·타입·백엔드 계약·MarkdownView·resolveWikiLinks·lineDiff**: 무변경.

## 데이터 흐름

```
편집 진입:  page.body (md) ──parseMarkdown()──▶ TipTap 문서
편집 중  :  TipTap 문서 상태만 변경 (마크다운 미개입 — 성능·커서 안정성)
저장     :  TipTap 문서 ──serializeMarkdown()──▶ updatePage/createPage({ title, body })
```

저장 이후(버전 생성, reloadPages, 보기 이동, 성공/실패 Toast)는 기존 흐름 그대로.

### [[위키링크]]

- 파싱: `[[제목]]` → 인라인 `wikiLink` 노드. 에디터 안에서 칩으로 렌더 —
  존재 페이지는 기본 칩, 부재 페이지는 danger 칩 (보기 화면의 파란/빨간 링크와 동일 의미론).
- 자동완성: `[[` 입력 시 같은 스페이스 페이지 후보 최대 8개, 키보드(↑↓/Enter/Escape) 탐색.
- 직렬화: `[[제목]]` 원문 텍스트로 복원 → 보기 화면 `resolveWikiLinks` 파이프라인 무변경.

### 손실 정책

- 에디터가 이해 못 하는 구문(생 HTML, 각주 등)은 일반 텍스트 문단으로 **보존** — 서식은 잃어도 내용은 삭제되지 않는다.
- 블록 화이트리스트 안에서 `serialize(parse(md)) ≒ md` (공백/개행 정규화 수준 차이만 허용)를 골든 테스트로 고정.
- **본문 불변 보장**: 편집 진입 시 원문을 보관하고, 본문을 한 번도 수정하지 않은 저장(제목만 변경 등)은
  직렬화 결과 대신 원문을 그대로 저장한다 — 불필요한 버전 diff 노이즈 방지.

### 편집 이탈 가드

dirty 상태(본문 또는 제목 변경)에서 취소·라우트 이탈 시 confirm("저장하지 않은 변경이 있습니다. 나가시겠습니까?").
저장 성공 후 가드 해제.

> 구현 노트(2026-07-17): 앱이 선언형 `<Routes>`를 사용해 `useBlocker`를 쓸 수 없다.
> 가드 범위는 ① 취소 버튼 confirm ② beforeunload(새로고침/닫기)로 한정하고,
> 사이드바 내비게이션 가드는 데이터 라우터 전환 시 후속 티켓으로 다룬다.

## 에러 처리

| 상황 | 처리 |
| --- | --- |
| 저장 실패 | 기존 패턴 — Toast(danger)에 `error.message`, 편집 내용 유지 |
| parseMarkdown 예외 | 원문 전체를 플레인 문단으로 로드하는 폴백 + 콘솔 경고 — 편집이 막히지 않게 |
| 자동완성/슬래시 메뉴 | 후보 0개 → 미표시, Escape·blur → 닫기 (기존 WikiLinkTextArea 규칙 이식) |
| 이미지 URL 로드 실패 | 대체 텍스트 + placeholder 박스 (깨진 이미지 노출 금지) |

## 테스트 전략

jsdom contenteditable 제약을 고려해 레이어 분리. 기존 스타일(vitest + Testing Library) 유지.

1. **`markdown.ts` 골든 왕복 테스트** (유닛, 가장 두껍게) — 블록 화이트리스트 전체.
   표·체크박스·중첩 목록·코드 블록 언어 태그·`[[링크]]` 포함. 데이터 안전의 핵심.
2. **wikiLink 확장 유닛** — `[[제목]]` ↔ 노드, 존재/부재 판별, 제목 제외 규칙.
3. **슬래시 메뉴·자동완성 로직 유닛** — 쿼리 감지·필터·키보드 선택을 TipTap 비의존 순수 함수로 분리해 테스트.
4. **App 통합 (씬 슬라이스)** — 편집 진입→입력→저장→보기 반영, dirty 이탈 confirm, 제목만 수정 시 본문 바이트 불변.
   드래그 핸들·버블 툴바의 브라우저 의존 동작(좌표·포지셔닝)은 통합 테스트 제외, 로직만 유닛 커버.

### 기존 테스트 영향

`PageEditPage`의 textarea를 직접 조작하는 테스트는 새 에디터 API에 맞게 수정 필요:
`App.w2-edit`, `App.w4-autocomplete`, `App.w4-links`(편집 부분), `App.w2-view`(편집 진입 경유 시).
구현 계획에서 개별 태스크로 명시한다.
